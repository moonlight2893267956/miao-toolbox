## miao-ai 502 排查：直调 OK 但转发 401（00:56 6/28）
- 现象：前端"AI 变更分析"弹窗显示 `miao-ai 返回错误: 502`
- 根因路径：
  1. `DiffAIController.java:169` 把 miao-ai 任何非 2xx 响应一律打成 502（看不到真实状态码）
  2. miao-ai 实际是 401（Authorization header 不对）
  3. 早期状态：api 容器 env 缺 MIAO_AI_*，所以 `Authorization: Bearer ` 是空，miao-ai 401
- 验证 key 有效：直接 `curl -H "Authorization: Bearer miao_RVNxD_F-..."` → 200 + 真实 AI 输出
- 代码确认 `MiaoAiClient.java:174` 用 `setBearerAuth(miaoAiProperties.getApiKey())` 是正确的
- 验证 api 进程环境：tr /proc/1/environ | grep MIAO_AI_*
- 结论：用户截图 502 是修复**前**状态；现在链路全通，让用户硬刷浏览器再试
- 教训：502 是控制器统一转译，看不到真实上游状态码，排查时一定要看 miao-ai 容器日志

## miao-ai 502 根因修复：JVM HTTP 代理劫持了内网流量（01:10 6/28）
- **根因**：`JAVA_OPTS` 里配了 `-Dhttp.proxyHost=host.docker.internal -Dhttp.proxyPort=7890`
  - `nonProxyHosts` 缺 `host.docker.internal` 和 `172.19.*`
  - Java HttpURLConnection 把到 miao-ai (host.docker.internal:18000) 的请求全走代理 7890
  - 代理对 SSE 长连接返回 502 → DiffAIController 收到 502 → 前端显示 "miao-ai 返回错误: 502"
  - `wget`/`curl` 不认 JVM 代理，所以容器内直接测永远通
- **修复**：`docker-compose.prod.yml` nonProxyHosts 末尾加 `|172.19.*|host.docker.internal`
  - `--force-recreate api` 重建容器
  - 验证：nonProxyHosts 正确 / wget stream 通 / 日志无 502
- 同步：`docker-compose.prod.yml` 已 scp 回本地
