# PRD: 阿渺工具箱

## 0. 文档目的

本文档面向开发者（Wuxiangyi）及后续可能加入的协作者，定义阿渺工具箱第一阶段的范围与需求。PRD 以功能分组、FR 全局编号、假设内联标注的方式组织。下游架构设计与 Epic 拆分以本文档为基准。

技术栈：Spring Boot + React。数据库：MySQL。部署：Docker Compose。

## 1. 愿景

阿渺工具箱是一个自托管的 AI 工具集成门户，将翻译、文生图、文生语音等独立工具统一到一个带认证和防护的 Web 平台中。核心价值：**让 AI 工具可管、可控、可扩展**——用户一次登录即可访问所有工具，而 AI 接口的调用始终在服务端代理与防护之下，杜绝密钥泄露与接口盗刷。

第一阶段目标：搭建安全骨架——认证体系、接口防护层、工具接入框架。具体的业务工具（翻译、文生图等）在后续阶段按插件式接入。

## 2. 目标用户

### 2.1 核心任务

- 作为使用者，我希望一次登录就能使用所有 AI 工具，而不用记住每个工具的入口和账号
- 作为管理者，我希望所有 AI 调用都经过我的后端，防止密钥泄露和被盗刷
- 作为管理者，我希望新增工具时只需接入标准接口，不用每次从零搭认证和防护

### 2.2 非用户（v1）

- 匿名访客——v1 不开放公开注册，所有使用者必须有账号
- 移动端原生用户——v1 仅提供 Web 界面，不做原生 App

### 2.3 关键用户旅程

- **UJ-1. 用户登录并使用一个 AI 工具**
  - **角色 + 上下文：** 阿渺的日常用户，已拥有账号，需要快速完成一次 AI 调用
  - **入口状态：** 未登录，浏览器打开阿渺工具箱首页
  - **路径：** 选择登录方式（账密 / GitHub OAuth）→ 完成认证 → 进入工具列表页 → 点击目标工具 → 输入内容 → 提交 → 获得结果
  - **高潮：** 结果在页面内直接展示，无需跳转或额外下载
  - **结束状态：** 停留在工具结果页，可继续使用或切换工具

- **UJ-2. 管理员监控异常调用**
  - **角色 + 上下文：** Wuxiangyi，收到异常告警或日常巡检
  - **入口状态：** 已登录管理员账号
  - **路径：** 进入管理后台 → 查看调用日志/仪表盘 → 定位异常用户或 IP → 查看详情 → 决定处置（禁用/限流）
  - **高潮：** 在管理界面直接完成处置操作，异常调用立即被阻断
  - **结束状态：** 异常已处理，系统恢复正常

## 3. 术语表

- **工具 (Tool)** — 阿渺工具箱中集成的一个独立 AI 功能单元（如翻译、文生图）。每个工具对应后端的一个接入模块和前端的一个操作界面。
- **工具接入框架 (Tool Integration Framework)** — 定义工具如何注册到平台的标准接口与协议，包括路由、参数校验、AI 调用代理、结果回调。
- **AI 代理层 (AI Proxy)** — 后端统一处理 AI 请求的中间层，负责密钥管理、请求转发、响应缓存、配额计量。
- **调用配额 (Quota)** — 用户在指定时间窗口内可使用的 AI 调用次数或资源量。[ASSUMPTION: v1 仅做速率限制，不按工具细分配额，具体配额策略留到业务阶段]
- **速率限制 (Rate Limiting)** — 对单用户/单 IP 在时间窗口内的请求频率上限，用于防刷而非计费。
- **管理员 (Admin)** — 拥有系统管理权限的用户角色。v1 通过数据库 `role` 字段标记，首个管理员在系统初始化时由迁移脚本创建。

## 4. 功能

### 4.1 用户认证

**描述：** 用户通过账密或 GitHub OAuth 登录，系统颁发会话凭证，后续所有请求均需携带有效凭证。实现 UJ-1。

**功能需求：**

#### FR-1: 账密注册与登录

用户可通过用户名 + 密码注册账号并登录。实现 UJ-1。

**可验证结果：**
- 用户名规则：3-20 位，字母数字下划线，唯一性校验
- 注册时密码强度校验：不少于 8 位，包含字母和数字
- 密码使用 bcrypt 加盐哈希存储，禁止明文或简单哈希（MD5/SHA256）
- 登录成功后颁发 JWT access token + httpOnly refresh token
- 连续 5 次登录失败后锁定账号 15 分钟

**不在范围内：**
- 邮箱验证流程（v1 可选）

#### FR-2: GitHub OAuth 登录

用户可通过 GitHub 第三方账号登录。[ASSUMPTION: v1 仅支持 GitHub OAuth；微信 OAuth 因需要企业资质，后续接入]

**可验证结果：**
- GitHub OAuth 授权流程完成即可创建/关联账号
- 首次 OAuth 登录自动创建本地账号，无需额外注册步骤
- 已有账密账号可绑定 GitHub，绑定后两种方式均可登录

#### FR-3: 会话管理与安全

系统对已认证会话进行安全管理。

**可验证结果：**
- Access token 有效期 15 分钟，refresh token 有效期 7 天
- Refresh token 轮换：每次刷新后旧 token 立即失效
- 单用户最多同时 5 个有效 refresh token，超出则最旧的失效
- 支持手动注销，注销后当前 session 的所有 token 立即失效
- 用户被禁用时，鉴权中间件拦截其所有请求（基于数据库用户状态实时校验）[ASSUMPTION: v1 在 JWT 校验时附加数据库用户状态查询，性能可接受；用户规模增长后引入 Redis 状态缓存]

#### FR-4: 传输加密

生产环境下所有前后端通信必须走 HTTPS。

**可验证结果：**
- 生产环境强制 HTTPS，HTTP 请求自动 301 重定向到 HTTPS
- 开发环境允许 HTTP（不强制 HTTPS）
- 使用 Let's Encrypt 证书，自动续期

### 4.2 接口防护

**描述：** 所有前端到后端的请求、以及后端到 AI 服务的请求，均经过统一防护层，防止盗刷、重放、绕过。实现 UJ-1、UJ-2。

**公开端点（无需鉴权）：** `/api/auth/login`、`/api/auth/register`、`/api/auth/oauth/**`、`/api/auth/refresh`。其余所有 API 均需鉴权。

**功能需求：**

#### FR-5: 统一 API 网关鉴权

所有业务 API 请求必须携带有效 JWT，网关层统一校验，未认证请求返回 401。

**可验证结果：**
- 未携带 token 或 token 过期的请求返回 401
- 无需在每个业务 Controller 中重复写鉴权逻辑
- 静态资源请求不经过鉴权
- 用户被禁用时，即使 token 未过期，请求也被拦截

#### FR-6: 速率限制

对认证用户按用户维度限流，对公开端点按 IP 维度限流。

**可验证结果：**
- 认证用户：单用户每分钟最多 60 次请求[ASSUMPTION: 初始阈值 60/min，通过 application.yml 配置，修改需重启]
- 公开端点：单 IP 每分钟最多 10 次请求（保护登录/注册不被暴力攻击）
- 超出限制返回 429 Too Many Requests，响应头包含 `Retry-After`
- 限流基于滑动窗口算法

#### FR-7: 请求防重放

API 请求携带时间戳与签名，服务端校验时效性与签名一致性。

**可验证结果：**
- 登录成功后，后端为该会话签发一个 `signingKey`，前端存储在内存中（不持久化到 localStorage）
- 请求必须携带 `X-Timestamp`（毫秒级）、`X-Nonce`（UUID）和 `X-Signature`（HMAC-SHA256，以 signingKey 为密钥，对 `timestamp + nonce + requestBody` 签名）
- 时间戳与服务器时间偏差超过 5 分钟的请求被拒绝
- 相同 nonce 的请求在窗口内只处理一次（按用户维度隔离）[ASSUMPTION: nonce 记录存 Redis，TTL 5 分钟]
- Token 刷新时 signingKey 同步更新，旧 signingKey 在 30 秒过渡期内仍可验证

#### FR-8: CORS 与来源校验

仅允许来自合法前端的跨域请求。

**可验证结果：**
- CORS 白名单仅包含已配置的前端域名（独立域名部署）
- 非白名单来源的请求被浏览器拦截或服务端拒绝
- 生产环境禁止 `Access-Control-Allow-Origin: *`

#### FR-9: 错误信息安全

所有面向前端的错误响应不泄露内部实现细节。

**可验证结果：**
- 500 错误返回通用消息（"服务器内部错误"），不包含堆栈信息、SQL 语句或服务拓扑
- AI 服务的原始错误码被后端统一包装后返回前端
- 日志中记录完整错误信息供排查，但前端不可见

### 4.3 AI 代理层

**描述：** 后端统一持有 AI 服务密钥，作为代理层转发所有 AI 请求。前端永远不直接接触 AI 服务的 API。实现 UJ-1。

**功能需求：**

#### FR-10: AI 密钥服务端托管

所有第三方 AI 服务的 API Key 仅存在于服务端环境变量或加密配置中，永不暴露给前端。

**可验证结果：**
- 前端代码、网络请求中不出现任何第三方 AI 服务的 API Key
- AI Key 通过环境变量注入，不硬编码在源码或配置文件中
- [ASSUMPTION: 生产环境使用 Docker secrets 或类似机制管理密钥]

#### FR-11: AI 请求代理

前端提交的 AI 调用请求由后端代理转发到对应的 AI 服务，响应原路返回。

**可验证结果：**
- 前端仅调用 `/api/tools/{toolId}/execute` 这类统一入口
- 后端根据 toolId 路由到对应 AI 服务，注入密钥，转发请求
- 代理层记录每次调用的用户、工具、耗时、token 消耗（如可获取）

#### FR-12: 输入输出安全

对用户提交给 AI 工具的输入进行基础安全过滤，对 AI 返回内容进行安全渲染。

**可验证结果：**
- 请求体大小限制：文本类 ≤ 10KB，文件类 ≤ 20MB[ASSUMPTION: 初始值，通过 application.yml 配置，后续按业务调整]
- 文件类型白名单校验（如仅允许 jpg/png/pdf）
- 输入内容经过 HTML 转义后再进入处理管道
- AI 返回内容在前端渲染前进行 XSS 过滤（使用 DOMPurify 或同类库），禁止直接 innerHTML 渲染
- v1 不做 Prompt 注入检测，仅做输入大小和类型限制[ASSUMPTION: Prompt 注入检测误报率高且规则维护成本大，v1 不引入，后续有业务需求时再评估]

### 4.4 工具接入框架

**描述：** 定义工具如何注册到平台的标准协议，使新增工具只需实现约定的接口即可接入，无需重复搭建认证和防护。实现 UJ-1。

**功能需求：**

#### FR-13: 工具注册协议

每个工具通过声明式配置注册到平台，包含元数据、参数定义、AI 服务对接方式。

**可验证结果：**
- 工具配置包含：ID、名称、描述、图标、参数 schema、AI 服务类型（内置代理 / 外部 API）、路由路径
- 工具注册后自动出现在用户可见的工具列表中
- 未注册的工具路径不可访问
- 配置文件格式为 YAML，示例如下：
  ```yaml
  tools:
    - id: text-translate
      name: 文本翻译
      description: 多语言文本翻译工具
      icon: translate
      route: /translate
      aiService: proxy          # proxy = 通过内置代理层
      params:
        - name: text
          type: string
          required: true
          maxLength: 5000
        - name: targetLang
          type: string
          required: true
  ```

#### FR-14: 工具执行管道

工具调用经过统一的执行管道：认证 → 限流 → 参数校验 → AI 代理 → 结果返回。

**可验证结果：**
- 所有工具调用走同一管道，新增工具无需重写鉴权/限流逻辑
- 参数校验基于工具声明的 schema 自动执行
- 管道采用 Spring HandlerInterceptor 责任链模式实现，各环节可独立插拔

#### FR-15: 工具管理界面

管理员可在后台查看已注册工具的状态，启用/禁用工具。

**可验证结果：**
- 管理后台展示工具列表：名称、状态、近期调用量
- 管理员可启用/禁用工具，禁用后用户不可见不可调用
- [ASSUMPTION: v1 工具注册通过配置文件完成，不提供 UI 动态注册；动态注册留到后续阶段]

### 4.5 管理与监控

**描述：** 管理员可查看系统运行状态、调用日志，并对异常情况进行处置。实现 UJ-2。

**功能需求：**

#### FR-16: 调用日志

系统记录所有 AI 工具调用的审计日志。

**可验证结果：**
- 日志包含：时间、用户 ID、工具 ID、请求摘要、响应状态、耗时、token 消耗（如可获取）
- 日志中不记录用户密码、完整请求体等敏感信息
- 日志保留至少 30 天
- 日志不可被普通用户访问

#### FR-17: 管理仪表盘

管理员可在后台查看系统概览。

**可验证结果：**
- 展示：今日总调用量、各工具调用量分布、近期异常请求趋势
- 展示：当前在线用户数、速率限制触发次数
- 页面手动刷新获取最新数据[ASSUMPTION: v1 不做实时推送，手动刷新即可]
- [ASSUMPTION: v1 仪表盘数据来自直接查询日志表，不做独立数据仓库；数据量增大后再考虑]

#### FR-18: 异常处置

管理员可对异常用户进行处置。

**可验证结果：**
- 管理员可禁用指定用户账号，禁用后该用户所有请求被鉴权中间件拦截（基于数据库状态实时校验）
- 管理员可对指定用户设置更严格的限流策略
- 处置操作本身被记录在审计日志中

### 4.6 管理员角色

**描述：** 系统区分管理员与普通用户角色，管理员拥有后台操作权限。

**功能需求：**

#### FR-19: 管理员身份

管理员通过数据库 `role` 字段标识，拥有管理后台的访问和操作权限。

**可验证结果：**
- 用户表包含 `role` 字段（`ADMIN` / `USER`）
- 系统初始化时通过迁移脚本创建首个管理员账号
- 管理员可访问 `/admin/**` 路径下的所有管理功能
- 普通用户访问 `/admin/**` 路径返回 403
- 管理员可升级/降级其他用户的角色

## 5. 非目标（明确声明）

- 不是 AI 模型训练或微调平台
- 不做工具间的数据联动或工作流编排（v1）
- 不做多租户隔离（v1 所有用户共享同一套工具配置）
- 不做计费系统（v1 无付费逻辑）
- 不做移动端原生应用
- 不做开放 API 供第三方调用（v1 仅服务自有前端）
- 不做 AI 异步任务回调（v1 仅支持同步请求-响应模式）

## 6. MVP 范围

### 6.1 纳入

- 账密注册/登录 + GitHub OAuth 登录
- 密码 bcrypt 加盐哈希存储
- JWT 会话管理（access + refresh token）
- HTTPS 强制（生产环境）
- 统一 API 网关鉴权 + 用户状态实时校验
- 速率限制（用户维度 + 公开端点 IP 维度）
- 请求防重放（HMAC 签名 + nonce）
- CORS 白名单（前端独立域名）
- 错误信息安全（不泄露内部细节）
- AI 密钥服务端托管
- AI 请求代理转发
- 输入大小/类型限制 + 输出 XSS 过滤
- 工具注册协议（YAML 配置）+ 执行管道（责任链模式）
- 管理员角色（数据库 role 字段）
- 管理员工具启用/禁用
- 调用审计日志（含脱敏）
- 管理仪表盘
- 异常用户处置（禁用、限流）
- Docker Compose 一键部署
- 前端：登录页、工具列表页、工具操作页、管理后台

### 6.2 不纳入 MVP

- 微信 OAuth 登录（需企业资质，后续接入）
- Prompt 注入检测（误报率高，后续评估）
- 按工具细分的调用配额系统（业务阶段再做）
- 工具 UI 动态注册（v1 通过配置文件注册）
- 独立日志数据仓库
- 邮箱验证流程
- 通知/告警推送（v1 依赖人工巡检仪表盘）
- AI 异步任务回调
- 实时数据推送（仪表盘手动刷新）

## 7. 成功指标

**首要指标**
- **SM-1**: 所有 AI 调用 100% 经过后端代理层，前端零直接调用。验证 FR-10, FR-11。

**次要指标**
- **SM-2**: 未认证/超限请求拦截率 100%（无漏网请求到达 AI 服务）。验证 FR-5, FR-6。
- **SM-3**: 新增一个工具从开发到可用的接入时间 ≤ 2 小时（不含 AI 服务对接本身）。验证 FR-13, FR-14。

**反向指标（不要优化）**
- **SM-C1**: 请求拦截数量——拦截多不代表系统更安全，可能是限流阈值设置过低。反向平衡 SM-2。
